# Scan Diagnostic — Why 0/0 Controls

**Date:** 2026-05-01
**Scan analyzed:** Most recent EU AI Act scan on Render

---

## TL;DR

The EU AI Act framework plugin has 7 rules. The scan engine evaluated all 7 in memory. Zero results were saved. The report was built from an empty database and reported 0/0. **The bug is that `Control` rows don't exist in the database for the EU AI Act framework, so the worker silently discards every evaluation result.**

## Phase 3 — The silent discard: root cause

### How control results are saved (worker route, `saveControlResult`):

```ts
async function saveControlResult(scanId, orgId, frameworkType, rule, raw) {
  const control = await db.control.findFirst({
    where: {
      code: rule.code,
      framework: { orgId, type: frameworkType },
    },
  });

  if (control) {
    await db.scanControlResult.upsert({ ... });
  }
  // ← NO else branch. If control is null: silent drop. No log. No error.
}
```

### How the post-phase builds the report:

```ts
// processPostPhase:
const scanResults = await db.scanControlResult.findMany({ where: { scanId } });
// If scanResults = [], then controlResults = []
const report = await buildReport(frameworkType, [], evidence);
// buildReport: controlsTotal: results.length → 0
//              controlsPassed: 0
```

### The chain of failure:

```
plugin.rules.length = 7 ✅
│
├─ processEvidencePhase: state.totalControls = 7 ✅
│
├─ processControlsPhase: for each of 7 rules:
│    ├─ evaluateControlWithLLM(rule, evidence) → produces result ✅
│    └─ saveControlResult(scanId, orgId, "EU_AI_ACT", rule, raw)
│         ├─ db.control.findFirst({ code: "AI-Art5", framework: EU_AI_ACT }) → NULL ❌
│         └─ [if (control)] skipped. Zero rows written to ScanControlResult. ❌
│
└─ processPostPhase:
     ├─ db.scanControlResult.findMany({ scanId }) → [] (0 rows) ❌
     ├─ buildReport("EU_AI_ACT", [], evidence) → { controlsTotal: 0, controlsPassed: 0 } ❌
     └─ Executive summary generated by Claude from 0/0 data (real text, wrong counts) ✅ (Claude improvises)
```

### Why are there no Control rows?

`Control` rows are **only created by one code path**: `POST /api/onboarding/questionnaire`.

```ts
// app/api/onboarding/questionnaire/route.ts
for (const fwType of classifier.applicableFrameworks) {
  const framework = await db.framework.upsert(...);
  const plugin = frameworkRegistry.get(fwType);
  if (plugin) {
    for (const rule of plugin.rules) {
      await db.control.upsert({ ... });  // ← ONLY place Controls are created
    }
  }
}
```

The project creation flow (`POST /api/projects`) creates `Framework` records but **does NOT create `Control` records**:

```ts
// app/api/projects/route.ts
await db.framework.upsert({
  where: { orgId_type: { orgId, type: fwType } },
  create: { orgId, type: fwType, status: "NOT_STARTED", score: 0, projectId: project.id },
  update: { projectId: project.id },
});
// ← No Control rows created here
```

**Most likely scenario:** The EU AI Act framework was created through the project creation flow, or onboarding was skipped. Either way, the `Control` table has 0 rows for EU AI Act. Confirm this with query #4 in `audit/scan-diagnostic.sql`.

## Phase 4 — Shadow pass / cross-framework (the 25% GDPR)

`runShadowPass()` does NOT read from the database at all:

```ts
export async function runShadowPass(evidence, excludeFramework) {
  for (const [key, plugin] of frameworkRegistry.entries()) {
    if (key === excludeFramework) continue;
    for (const rule of plugin.rules) {
      const result = runControl(rule, evidence);  // In-memory, pure function
      if (result.status === "PASS") met++;
    }
  }
}
```

It runs each framework's rule `check()` functions directly against the evidence pool. The GDPR 25% (2/8 rules passing) is **derived from real evidence** — either GitHub signals, onboarding answers (usesAI, dataCategories), or inferred questionnaire values. It is not fabricated or hardcoded.

**This means:** shadow pass works correctly even with 0 DB rows for the primary framework. The 25% GDPR is a real inference. The EU AI Act 0% is not — it reflects the missing DB rows, not the actual evaluation.

## Phase 6 — Observability gap

The worker route has **one** `console.error` statement (in the catch block). The scan engine (`lib/scan-engine/index.ts`) has **zero** `console.log` statements.

Specifically, `saveControlResult` has no logging whatsoever. When it silently drops a control result, nothing appears in Render logs. The scan completes successfully (HTTP 200, status = COMPLETED) and the only evidence of the failure is a 0/0 in the report.

This means:
- You cannot diagnose this from Render logs alone
- You cannot diagnose this from QStash delivery logs (they show 200 OK)
- The only diagnostic signal is the DB state — hence these SQL queries

---

## Top finding: Control rows are never created for project-flow frameworks

**Root cause in one sentence:** `saveControlResult()` silently discards all 7 EU AI Act evaluations because no matching `Control` rows exist in the database, and the report is built from 0 database rows instead of 7 in-memory results.

**Why it's silent:** The `if (control) { upsert }` pattern has no `else` — no error, no log, no counter. The scan reports `{ ok: true }` for every chunk and completes with `status: COMPLETED`. The 0/0 only appears in the report UI.

---

## Other findings (priority ordered)

1. **`saveControlResult` has no self-healing.** It never creates the `Control` row it can't find. Fix: upsert the `Control` row if missing, or pre-create all `Control` rows before the controls phase begins. Either approach ends the silent discard.

2. **Project flow and onboarding flow are diverged.** Onboarding creates `Framework` + `Control` rows. Project creation creates `Framework` rows only. Any framework added through a project (not onboarding) will always produce 0/0. This affects ALL frameworks a user adds through the project UI, not just EU AI Act.

3. **GitHub data requires a manual sync.** Connecting GitHub does not auto-scan the repo. The auto-sync only runs for Google Workspace, Slack, and Notion. A first-time GitHub user will always have empty `codeSignals.github` unless they navigate to the integrations page and trigger a scan. This should be surfaced in the UI.

4. **Zero logging in the controls evaluation path.** Without logs, every future 0/0 will require running SQL queries to diagnose. The fix prompt should add at least one `console.log` in `saveControlResult` for the null-control case.

5. **The executive summary is generated from 0 data.** Claude produces coherent text about EU AI Act fines and Annex III even when `controlsPassed: 0, controlsTotal: 0`. This means the summary is generic boilerplate, not derived from actual scan findings. It will always look plausible even when the scan is completely broken. Users may not realize the report is invalid.

## What to do next

**Do not attempt to fix the 7 individual rules** — they are fine. Do not add more rules. Do not touch the scan engine logic. Do not re-run the scan (it will produce the same result).

**The fix is in one of two places** (user decides which approach after seeing the SQL output):

**Option A — Fix `processEvidencePhase` to pre-create Control rows** (preferred):  
Before queuing the first controls chunk, upsert all `Control` rows for the framework. The framework record already exists at that point (`db.framework.findFirst({ orgId, type })`). This is a ~15-line addition to the worker route's evidence phase.

**Option B — Fix `saveControlResult` to upsert the Control row if missing** (quicker but messier):  
When `db.control.findFirst` returns null, look up the framework record and create the control first. Messier because it requires an extra DB round-trip per control per scan, but it's self-healing.

**After the fix:** Every new scan will work. Existing scans with 0/0 will need to be re-run. Old `Scan` records with bad `reportJson` won't auto-fix — the user would need to trigger a new scan on the same framework.

The next prompt should focus **only on this fix**. Don't bundle cleanup, logging improvements, or GitHub UX changes into the same PR.
